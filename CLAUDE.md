# CLAUDE.md

## Project Overview
Daphnia is an open-source SVG to CustomPainter code generator for Dart/Flutter.

**Goal:** Users annotate an SVG file, and the code generator produces a CustomPainter class that draws the shape with customizable color variables.

**Target:** Dart 3.10.0 / Flutter 3.38.0 (managed via FVM)

**Publishing:** pub.dev (open source, hosted on GitHub)

## Architecture

This is a code generation package using `source_gen` and `build_runner`. Structure:
- `packages/daphnia_annotation` - Contains the `@SvgPainter` annotation
- `packages/daphnia` - The code generator that produces CustomPainter classes
- `example/` - Example Flutter app demonstrating usage

## Development Principles

### Design Principles
We follow S.O.L.I.D. combined with YAGNI, DRY, and KISS:
- **S** - Single Responsibility Principle: A class should have only one reason to change
- **O** - Open/Closed Principle: Open for extension, closed for modification
- **L** - Liskov Substitution Principle: Subtypes must be substitutable for base types
- **I** - Interface Segregation Principle: Many small interfaces over one large one
- **D** - Dependency Inversion Principle: Depend on abstractions, not concretions
- **YAGNI** - Only implement what's needed now
- **DRY** - No duplicate code
- **KISS** - Keep it simple

### Coding Style - Linting
We use strict linting rules from the Flutter repository. No analysis issues allowed (errors, warnings, or infos). Strict casting and inference enabled.

### Coding Style - Conditionals
Always use positive conditions. Avoid negations (`!`) in conditional statements for readability.

### Coding Style - Parameters
Functions with more than 2 parameters should use named parameters.

### Exception Handling
Use the `Either<Failure, Success>` pattern when something can fail. Never let functions throw exceptions.

## Testing

### AAA Pattern
All unit tests follow Arrange-Act-Assert:
```dart
test('should do something', () {
  // arrange
  final input = 'test';
  // act
  final result = doSomething(input);
  // assert
  expect(result, expectedValue);
});
```

### File Structure
Test files map 1:1 to source files:
- Source: `lib/src/parser/color.dart`
- Test: `test/src/parser/color_test.dart`

### Group Naming
- Constructors: `group('ClassName()', () { ... })`
- Methods: `group('ClassName.methodName()', () { ... })`

### Const Classes
Const classes require two tests:
1. **Canonicalization** - Verify identical const instances (use `const`)
2. **Field assignment** - Verify fields store provided values (use `final` for coverage)

### Enums
Do NOT test canonicalization for enums - it's guaranteed by Dart.

### Test Object Types
- **Mocks** - Generated by mockito with `@GenerateNiceMocks`
- **Stubs** - Manual, in `*_test.classes.dart`, prefix `Stub`
- **Fakes** - Manual, in `*_test.classes.dart`, prefix `Fake`
- **Dummies** - Manual, in `*_test.classes.dart`, prefix `Dummy`
